# 0.1.1 — 2026-06-27

First release. An authoring-loop MCP server that turns an agent into an eval author: the agent produces a verifiable eval record through a draft → review → surgical-revise → submit loop, and the server acts as both scribe (normalize, persist, compile) and adversarial checker (run the record's own grader, reject what doesn't hold up). No external API — records are plain JSON files under `EVALS_DATA_DIR`. The package is scoped `@cyanheads/evals-mcp-server`.

## Added

- **`evals_describe_schema`** — required/optional fields plus grader options for a `task_type` (static, derived from the schemas).
- **`evals_create_draft`** — create a draft carrying its own executable grader; reflects it back with a per-field review protocol, a verification subagent prompt, and a self-consistency verdict.
- **`evals_get_record`** — read a draft or submitted record by id; the verification subagent's entry point.
- **`evals_revise_draft`** — surgical `set` / `append` / `unset` patch to a draft by dotted path, re-running the self-consistency check.
- **`evals_discard_draft`** — delete a draft (draft-only).
- **`evals_run_check`** — run a grader against candidates for a PASS/REJECT verdict plus the resolved comparison value, decoupled from any saved record.
- **`evals_submit_draft`** — committability gate, then freeze.
- **`evals_list_records`** — filter records by status / domain / task_type / tag, returning a compact summary projection.
- **`evals_export_records`** — compile submitted records to JSONL, CSV, Inspect AI, or lm-evaluation-harness under `exports/`.
- **`eval://record/{id}` resource** — a record mirror of `evals_get_record` for resource-capable clients.
- **Record schema** — a Zod `discriminatedUnion` on `task_type` (`numeric`, `exact_answer`, `set_answer`, `mcq`, `regex_answer`, `json_answer`, `free_response`).
- **Grader DSL** — a typed union serialized with each record; deterministic kinds (`numeric` via math.js, `exact_match`, `set_match`, `regex`, `mcq`, `json_match`) run server-side, `llm_rubric` routes to sampling.
- **Committability gate** at submit — the gold must pass its own grader, ≥1 declared negative case must be rejected, and a recorded, decorrelated independent verification must agree with the gold.
- **Provenance capture** — link framework-written tool-call dumps via the `captures` EvalsID field, resolved from `EVALS_CAPTURE_DIR` and cross-checked against the gold.
- **Deployment** — STDIO and Streamable HTTP transports, an `.mcpb` bundle for Claude Desktop, a Dockerfile for container builds, and Codex / Claude Code plugin manifests.
